# Code Documentation: AI Multi-Agent Research Assistant

This document explains how `app.py` works and how each class/function contributes to the final output shown in the Streamlit app or returned by the FastAPI endpoint.

## 1. Overall Application Flow

```text
User Query
  -> Streamlit UI or FastAPI endpoint
  -> run_research()
  -> LangGraph workflow
  -> planner_agent()
  -> researcher_agent()
  -> writer_agent()
  -> evaluator_agent()
  -> Final Answer
```

The application uses:

- **Streamlit** for the user interface.
- **FastAPI** for backend API access.
- **LangGraph** to manage the multi-agent workflow.
- **LangChain** for prompts, LLM calls, tools, document loading, and retrieval.
- **Groq API with Llama 3.3** for fast LLM inference.
- **DuckDuckGo Search** for web research.
- **HuggingFace Embeddings** for semantic vector embeddings.
- **ChromaDB** for vector storage and retrieval.
- **MongoDB Atlas** for optional session persistence.

## 2. Configuration Constants

### `APP_TITLE`

Stores the application name:

```python
APP_TITLE = "AI Multi-Agent Research Assistant"
```

Used by both Streamlit and FastAPI.

### `DEFAULT_MODEL`

Loads the Groq model name from the environment:

```python
DEFAULT_MODEL = os.getenv("GROQ_MODEL", "llama-3.3-70b-versatile")
```

If `GROQ_MODEL` is not set, the app uses `llama-3.3-70b-versatile`.

### `CHROMA_DIR`

Defines where ChromaDB stores vector data:

```python
CHROMA_DIR = os.getenv("CHROMA_DIR", "./chroma_db")
```

### `EMBEDDING_MODEL`

Defines the HuggingFace embedding model:

```python
EMBEDDING_MODEL = os.getenv("EMBEDDING_MODEL", "sentence-transformers/all-MiniLM-L6-v2")
```

## 3. Data Classes and Schemas

### `ResearchState`

```python
class ResearchState(TypedDict, total=False):
```

This is the shared state object passed between LangGraph agents.

It stores:

- `session_id`: Unique ID for each research run.
- `query`: User research question.
- `documents`: Uploaded documents loaded as LangChain `Document` objects.
- `plan`: Research plan generated by the Planner Agent.
- `search_results`: Web results from DuckDuckGo.
- `rag_context`: Relevant retrieved document chunks from ChromaDB.
- `insights`: Extracted analysis from the Researcher Agent.
- `report`: Final answer generated by the Writer Agent.
- `evaluation`: Quality check generated by the Evaluator Agent.
- `progress`: Workflow progress messages.
- `errors`: Any errors during research. 
- `use_web`: Whether web search is enabled.
- `use_rag`: Whether document retrieval is enabled.
- `max_results`: Maximum DuckDuckGo results.

This state is important because every agent reads from it and writes back to it.

### `ResearchRequest`

```python
class ResearchRequest(BaseModel):
```

Defines the request body for the FastAPI `/research` endpoint.

Fields:

- `query`: Required user query.
- `use_web`: Enables or disables DuckDuckGo search.
- `use_rag`: Enables or disables document RAG.
- `max_results`: Number of search results, between 1 and 10.

### `ResearchResponse`

```python
class ResearchResponse(BaseModel):
```

Defines the API response returned by FastAPI.

Fields:

- `session_id`
- `query`
- `plan`
- `insights`
- `report`
- `evaluation`
- `progress`

The `report` field contains the final answer.

## 4. LLM and Storage Helper Functions

### `get_llm()`

```python
def get_llm(model: str = DEFAULT_MODEL, temperature: float = 0.2) -> ChatGroq:
```

Creates and returns a Groq LLM client.

Steps:

1. Checks whether `GROQ_API_KEY` exists.
2. Raises an error if the API key is missing.
3. Returns a `ChatGroq` model using the selected model and temperature.

Used by:

- `planner_agent()`
- `researcher_agent()`
- `writer_agent()`
- `evaluator_agent()`

### `get_embeddings()`

```python
def get_embeddings() -> HuggingFaceEmbeddings:
```

Creates the HuggingFace embedding model.

Used by:

- `build_vectorstore()`

Purpose:

- Converts text chunks into vectors.
- Enables semantic search in ChromaDB.

### `get_mongo_collection()`

```python
def get_mongo_collection():
```

Connects to MongoDB Atlas if `MONGODB_URI` is configured.

Steps:

1. Reads `MONGODB_URI`.
2. If missing, returns `None`.
3. Creates a MongoDB client.
4. Selects the configured database and collection.

Used by:

- `save_session()`

### `save_session()`

```python
def save_session(state: ResearchState) -> None:
```

Stores the completed research session in MongoDB.

It saves:

- Query
- Plan
- Insights
- Final answer
- Evaluation
- Progress logs
- Errors
- Update timestamp

If MongoDB is not configured, this function safely does nothing.

### `append_progress()`

```python
def append_progress(state: ResearchState, message: str) -> ResearchState:
```

Adds progress messages to the shared workflow state.

Example progress messages:

- `Planner Agent: creating research strategy.`
- `Researcher Agent: gathering web and document evidence.`
- `Writer Agent: drafting final answer only.`
- `Workflow complete.`

These messages are displayed in the Streamlit progress area.

## 5. Document Processing and RAG Functions

### `load_uploaded_documents()`

```python
def load_uploaded_documents(uploaded_files) -> List[Document]:
```

Loads user-uploaded files from the Streamlit uploader.

Supported file types:

- PDF
- TXT
- Markdown
- DOCX
- HTML
- Other formats supported by Unstructured

Loader selection:

- `.pdf` uses `PyPDFLoader`.
- `.txt` and `.md` use `TextLoader`.
- Other files use `UnstructuredFileLoader`.

Output:

- A list of LangChain `Document` objects.

Used by:

- `render_streamlit_app()`

### `build_vectorstore()`

```python
def build_vectorstore(session_id: str, documents: List[Document]) -> Optional[Chroma]:
```

Builds a ChromaDB vector store from uploaded documents.

Steps:

1. Returns `None` if no documents exist.
2. Splits documents into chunks with `RecursiveCharacterTextSplitter`.
3. Adds session metadata to each chunk.
4. Converts chunks into embeddings using HuggingFace.
5. Stores vectors in ChromaDB under a session-specific collection.

Purpose:

- Makes uploaded documents searchable by meaning, not exact keywords.

### `retrieve_context()`

```python
def retrieve_context(session_id: str, query: str, documents: List[Document], use_rag: bool) -> str:
```

Retrieves the most relevant document chunks for the user query.

Steps:

1. Skips retrieval if RAG is disabled.
2. Skips retrieval if no documents are uploaded.
3. Builds the Chroma vector store.
4. Performs similarity search using the query.
5. Returns the top matching chunks as text context.

Used by:

- `researcher_agent()`

## 6. Agent Functions

### `planner_agent()`

```python
def planner_agent(state: ResearchState) -> ResearchState:
```

Creates the research strategy.

Input:

- User query from `state["query"]`

Process:

1. Adds a progress message.
2. Creates a planning prompt.
3. Calls the Groq LLM.
4. Stores the result in `state["plan"]`.

Output:

- Updated state with a research plan.

Technology:

- LangChain prompt
- Groq Llama model
- LangGraph node

### `researcher_agent()`

```python
def researcher_agent(state: ResearchState) -> ResearchState:
```

Collects and analyzes research evidence.

Input:

- Query
- Plan
- Uploaded documents
- Web search settings
- RAG settings

Process:

1. Adds a progress message.
2. Uses DuckDuckGo Search if web search is enabled.
3. Retrieves document context using RAG if enabled.
4. Sends web evidence and document context to the Groq LLM.
5. Extracts relevant facts, themes, gaps, and insights.

Output:

- `state["search_results"]`
- `state["rag_context"]`
- `state["insights"]`

Technology:

- DuckDuckGo Search
- LangChain tools
- ChromaDB
- HuggingFace Embeddings
- Groq Llama model

### `writer_agent()`

```python
def writer_agent(state: ResearchState) -> ResearchState:
```

Creates the final user-facing answer.

Input:

- Query
- Research plan
- Research insights
- Source results

Current output rule:

```markdown
**Final Answer**
- Short relevant point
- Short relevant point
- Short relevant point
```

The Writer Agent is instructed not to write:

- Essay
- Long report
- Title
- Executive summary
- Methodology
- Conclusion
- Generic filler

Output:

- Stores final answer in `state["report"]`.

Technology:

- LangChain prompt
- Groq Llama model
- LangGraph node

### `evaluator_agent()`

```python
def evaluator_agent(state: ResearchState) -> ResearchState:
```

Checks the quality of the final answer.

Input:

- Query
- Plan
- Final answer

Process:

1. Adds a progress message.
2. Calls Groq LLM with an evaluation prompt.
3. Scores the answer on:
   - Relevance
   - Evidence use
   - Clarity
   - Completeness
4. Stores the result in `state["evaluation"]`.
5. Saves the session to MongoDB if configured.

Output:

- Updated state with evaluation feedback.

## 7. LangGraph Workflow Functions

### `build_graph()`

```python
def build_graph():
```

Creates the LangGraph workflow.

Nodes:

- `planner`
- `researcher`
- `writer`
- `evaluator`

Edges:

```text
planner -> researcher -> writer -> evaluator -> END
```

Purpose:

- Ensures each agent runs in the correct order.
- Passes `ResearchState` from one agent to the next.

### `run_research()`

```python
def run_research(query, documents=None, use_web=True, use_rag=True, max_results=5) -> ResearchState:
```

Starts a complete research session.

Steps:

1. Creates a unique `session_id`.
2. Builds the initial `ResearchState`.
3. Calls `build_graph()`.
4. Runs the full LangGraph workflow.
5. Returns the final state.

Used by:

- FastAPI `/research`
- Streamlit app button

This is the main function that produces the final answer.

## 9. Streamlit UI Function

### `render_streamlit_app()`

```python
def render_streamlit_app() -> None:
```

Builds the interactive web interface.

UI sections:

- Page title
- Sidebar configuration
- Query input box
- File uploader
- Run button
- Agent progress display
- Final answer display
- Debug/detail tabs

Sidebar controls:

- Groq model name
- Enable/disable DuckDuckGo search
- Enable/disable RAG
- Number of search results
- Document upload

Main workflow:

1. User enters a query.
2. User optionally uploads documents.
3. User clicks `Run Multi-Agent Research`.
4. Uploaded files are loaded with `load_uploaded_documents()`.
5. `run_research()` executes the LangGraph workflow.
6. The final answer is displayed under `Final Answer`.

Tabs:

- `Plan`: Planner Agent output.
- `Insights`: Researcher Agent output.
- `Evaluation`: Evaluator Agent output.
- `Sources`: DuckDuckGo search results.
- `Raw JSON`: Full internal state except uploaded document objects.

## 10. Application Entry Point

```python
if __name__ == "__main__":
    render_streamlit_app()
```

This runs the Streamlit app when the file is executed directly by:

```bash
streamlit run app.py
```

When FastAPI imports the file with:

```bash
uvicorn app:api_app
```

the Streamlit UI does not run. Only the FastAPI object is loaded.

## 11. Final Output Generation

The final output is created by this sequence:

```text
run_research()
  -> build_graph()
  -> planner_agent()
  -> researcher_agent()
  -> writer_agent()
  -> evaluator_agent()
```

The specific final answer shown to the user comes from:

```python
state["report"]
```

That value is generated inside:

```python
writer_agent()
```

and displayed in Streamlit through:

```python
st.markdown(state.get("report", ""))
```

## 12. Example Final Output

For a query like:

```text
What are AI agents?
```

The final output should look like:

```markdown
**Final Answer**
- AI agents are systems that use an LLM to reason, make decisions, and take actions.
- They can call tools such as search, databases, APIs, and document retrievers.
- A typical agent workflow includes planning, action, observation, memory, and evaluation.
- LangGraph is useful for coordinating multiple agents in a controlled workflow.
- In this app, the Planner, Researcher, Writer, and Evaluator agents work together to produce the final answer.
```

This is intentionally short and focused, not a long essay or formal research report.
